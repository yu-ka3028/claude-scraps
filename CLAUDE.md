# スクラップ単語帳 ルール定義

このディレクトリは、Claude Codeセッション中に出会った新しい概念・用語を記録し、
スペースドリピティション的に復習するための単語帳システムです。

---

## ファイル構成

```
Scraps/
├── CLAUDE.md                        ← このファイル（ルール定義）
├── _index.md                        ← 全スクラップの復習日・理解度を一元管理
├── _template.md                     ← 新規スクラップファイルのテンプレート
├── _archive/                        ← 理解度[x][x][x][x]になったスクラップの保管庫（Obsidianグラフには残す）
│   ├── SQL/
│   └── Web/
├── SQL/                             ← カテゴリ別フォルダ
│   ├── CTE.md                      ← スクラップ1件 = 1ファイル
│   └── ウィンドウ関数-OVER句.md
├── Web/
│   └── プロキシ.md
└── .claude/
    └── skills/
        ├── status-scrap/SKILL.md
        ├── review-scrap/SKILL.md
        └── report-scrap/SKILL.md
```

- **1スクラップ = 1ファイル**
- **カテゴリ別フォルダ**に保存する（`SQL/`, `Web/`, `Next/`, etc.）
- カテゴリフォルダが存在しない場合は新規作成する
- **`_index.md`** がすべてのスクラップの復習日・理解度を一元管理する

---

## ファイル命名規則

- 見出し名をベースに、括弧内の長い説明は省く
- 英語の固有名詞はそのまま使う（`CTE.md`, `bfcache.md`）
- 日本語・英語混在は問題ない
- スペースはハイフンに置き換える（`ウィンドウ関数-OVER句.md`）
- 記号（`()`, `[]`, `*`）は省く

---

## スクラップのフォーマット

各スクラップファイルは以下の形式で記述する:

```markdown
| 項目 | 値 |
|------|---|
| 理解度 | [x][ ][ ][ ] |
| 保存日 | YYYY-MM-DD |
| 最終復習 | YYYY-MM-DD |
| 次回復習 | YYYY-MM-DD |

**Q:** {質問文}

**A:** {回答}

**メモ:** {補足・背景・使用例など（省略可）}
```

---

## 理解度レベルと復習スケジュール

| インジケーター | 意味 | 次回復習まで |
|----------------|------|-------------|
| [x][ ][ ][ ] | 保存したばかり。まだ復習していない | 1日後 |
| [x][x][ ][ ] | 質問に対して概要を言えた | 3日後 |
| [x][x][x][ ] | 自分の言葉で説明でき、腑に落ちている | 7日後 |
| [x][x][x][x] | 完全に定着 → `_archive/` へ移動 | 出題しない |

理解度を更新したとき、**更新後のレベルに対応する日数**で次回復習日を設定する。

## Claudeへの指示（操作方法）

### スクラップを保存する

ユーザーが「スクラップに保存して」「これを残して」などと言ったとき:

1. カテゴリを判断（不明な場合はユーザーに確認）
2. セッション内の会話を振り返り、以下の内容を自動生成する:
   - ファイル名（命名規則に従う）
   - **Q:** そのトピックの核心を問う1文
   - **A:** 回答の要点（簡潔に）
   - **メモ:** 補足・背景・使用例（あれば）。既存スクラップと関連があれば `[[ファイル名]]` を自然な文脈で含める
3. **Q だけをユーザーに提示**して答えさせる
4. A を開示してフィードバックする
5. 以下の質問を投げかける:

   > 「{カテゴリ名}/{ファイル名} にこの内容で保存するなら理解度を1〜4で回答してください。」
   > ```
   > 1: [x][ ][ ][ ] 保存したばかり
   > 2: [x][x][ ][ ] 概要は言える
   > 3: [x][x][x][ ] 自分の言葉で説明できる
   > 4: [x][x][x][x] 完全に定着（即アーカイブ）
   > ```

6. ユーザーの回答を確認する:
   - **1〜4 が返ってきた場合**: 対応する理解度でファイルを作成し、`_index.md` を更新する
   - **それ以外が返ってきた場合**: 修正内容を反映して同じ質問を再度投げかける（1〜4が返るまで繰り返す）
7. 保存日・最終復習日は本日の日付、次回復習日は選択した理解度に対応する日数で設定する
8. 理解度4の場合は `_archive/{カテゴリ}/` へ移動（`_index.md` には追加しない）

### 復習する

`/review-scrap` スラッシュコマンドで実行する。詳細な手順は `.claude/skills/review-scrap/SKILL.md` に定義。

### レポートを生成する

`/report-scrap` スラッシュコマンドで実行する。詳細な手順は `.claude/skills/report-scrap/SKILL.md` に定義。
期間（week / 2weeks / month）を指定して、活動サマリー・理解度分布・基礎トピック進捗・次の学習提案を表示する。

### 理解度を更新する

- 理解度の判断は**ユーザー自身が行う**（Claudeは提案しない）
- 更新時に書き換える項目（**スクラップファイルと `_index.md` の両方**を更新）:
  - 理解度インジケーター
  - 最終復習日 → 本日の日付
  - 次回復習日 → 更新後のレベルに対応する日数を本日に加算

| 更新後レベル | 次回復習日 |
|-------------|-----------|
| [x][ ][ ][ ] | 本日 + 1日 |
| [x][x][ ][ ] | 本日 + 3日 |
| [x][x][x][ ] | 本日 + 7日 |
| [x][x][x][x] | アーカイブへ移動（`_index.md` から削除） |

- ユーザーが `[x][x][x][x]` と申告した場合:
  1. スクラップファイルを `_archive/{カテゴリ}/` へ移動する（`git mv`）
  2. `_index.md` から該当行を削除する
  3. 以降の復習には出題しない

### `_index.md` の管理

`_index.md` は以下の形式で全スクラップを管理する:

```markdown
# インデックス

| ファイル | 次回復習 | 理解度 |
|---------|---------|-------|
| SQL/CTE.md | 2026-03-06 | [x][x][ ][ ] |
| Web/プロキシ.md | 2026-03-08 | [x][x][x][ ] |
```

- スクラップの保存・理解度更新・アーカイブのたびに必ず更新する
- `status-scrap` / `review-scrap` はこのファイルのみを読んで対象を絞り込む
- **キーワード列**: 保存時にそのスクラップの主要概念を3〜5個カンマ区切りで記載する。保存時にキーワードが重複する既存スクラップのみを読み込んでリンク候補を判断することで、全件スキャンを避ける

### Git コミット・プッシュ

スクラップの**保存**または**理解度更新**を行ったとき、ファイル更新の直後に自動でコミット・プッシュする:

```bash
git add {更新したファイル} _index.md
git commit -m "scrap: {操作内容の要約} (YYYY-MM-DD)"
git push origin main
```

コミットメッセージの例:
- `scrap: add SQL/CTE (2026-02-25)`
- `scrap: review SQL, Web — update 3 items (2026-02-25)`
- `scrap: archive SQL/upsert-トグル設計 (2026-02-25)`

ユーザーへの確認は不要。ファイル保存と一体の操作として実行する。

---

## 注意事項

- 1セッションで複数のスクラップを保存する場合は、それぞれ別のファイルとして追加する
- カテゴリ名はなるべく簡潔に（例: `SQL`, `TypeScript`, `HTTP`, `Git`, `設計`）
- メモ欄は省略可だが、当時の文脈や使用例があると復習時に役立つ
- スクラップ間のリンクは保存時にClaudeが判断してメモ欄に `[[ファイル名]]` を自然な文脈で埋め込む。Obsidianはそれをグラフに自動反映する
