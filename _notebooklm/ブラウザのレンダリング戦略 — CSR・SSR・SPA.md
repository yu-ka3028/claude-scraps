# ブラウザのレンダリング戦略 — CSR・SSR・SPA

Webアプリがどうやって画面を表示するかは、大きく2つに分かれます。クライアントレンダリングとサーバーレンダリングです。クライアントレンダリングは空のHTMLとJavaScriptをブラウザに送って、ブラウザ側でコンテンツを生成します。JavaScriptが動くまで画面が表示されないのが弱点です。サーバーレンダリングは逆で、サーバーがデータベースからデータを取得してHTMLを作り、完成品を送ります。表示が速く、DBへのアクセスキーもサーバーから外に出ないので安全です。

SPAはアーキテクチャの話で、最初に1枚のHTMLを受け取り、以後はJavaScriptがルーティングしてページ全体のリロードなしにDOMを書き換えます。Rails ERBのようにページ遷移のたびにHTMLが返ってくるMPAとは対照的です。SPA・MPAはアーキテクチャで、CSR・SSRはレンダリング手法という分類になります。SPAは通常CSRで実装されますが、この2つは別の概念です。

hydrationはSSRと切り離せない概念で、サーバーで生成された静的なHTMLにイベントリスナーをアタッチしてインタラクティブにする工程のことです。SSRは表示は速いのですが、ユーザーがクリックできるようになるまでにhydrationが完了する必要があります。React Server Componentsの大きな特徴の一つが、このhydrationを不要にした点で、JSをブラウザに送らないのでアタッチするものがそもそもないんですよね。
