# SQLの正規化と正規形

正規化って何をやってるのかというと、一言で言うと「同じ情報は一か所にしか書かない」という設計原則なんですよね。データが複数の場所に重複して存在すると、更新したときに片方だけ更新されてデータが矛盾する、いわゆる更新異常が起きます。正規化はこれを防ぐためのものです。

正規化には段階があって、第1・第2・第3正規形という3つのステップで整理していきます。第1正規形はまず「セルに値は1つだけ」というルール。カンマ区切りでIDを詰め込むのは、ここで既にアウトです。第2正規形は、複合主キーのテーブルで「キーの一部にしか依存しない列を別テーブルへ分離する」もの。注文明細テーブルに商品名が入っているのが典型例で、商品名は商品IDだけで決まるので商品テーブルへ出す必要があります。第3正規形はさらに厳しくて、「主キー以外の列を経由した依存」を排除します。社員テーブルに部署名が入っているケースがわかりやすくて、社員IDから部署IDを経由して部署名が決まる、この数珠つなぎが問題になります。

数字が大きいほど基準が厳しくなりますが、本質的な問いはどのレベルでも同じで、「この列、ここに置く必要ある？」という問いを繰り返すことなんですよね。
